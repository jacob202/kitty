     Phase 2 ‚ÄĒ Foundation rebuild, future-proofed

     Why this phase exists

     Memory is the heart of "Kitty knows me." The previous data was garbage; we
     wiped it in Phase 1. Now we rebuild ingestion correctly ‚ÄĒ schema-versioned,
      source-tagged, idempotent. And we instrument KittyBuilder for token cost
     so we can measure where money and time go before we start optimizing. And
     we align tool runtime so one skill's output can feed another's input
     cleanly.

     This is the longest phase. It has three subphases (2A, 2B, 2C) which can be
      worked in parallel by different sessions but must all converge before
     Phase 3.

     Definition of done

     - Memory architecture decision document committed (one page)
     - Schema for ingested records defined and versioned (schema_version field
     on every record)
     - Idempotent ingestion pipeline: same input ‚Üí same record, no duplicates
     - Retrieval test: known-good query returns expected record (round-trip)
     - KittyBuilder per-call token logging live (input tokens, output tokens,
     model, cost in $0.xxx)
     - Token analytics view: a scripts/kitty_token_analytics.sh that prints top
     costs by model and skill
     - All 8 missing KittyBuilder test cases (named below) added and passing
     - Tool runtime: defined input/output/error shape; bash/jq/awk wrappers as
     deterministic tools; one tool chain end-to-end

     Subphase 2A ‚ÄĒ Memory architecture (rebuild)

     Architectural decision required upfront: Should we keep the LightRAG +
     ChromaDB + SQLite-vec stack, or simplify to one or two? My recommendation
     is simplify to two: SQLite (with sqlite-vec extension for embeddings) for
     journal entries + structured records; ChromaDB only if we genuinely need
     its query semantics for unstructured KB (we may not). Drop LightRAG unless
     a specific feature of it is load-bearing ‚ÄĒ three vector stores is
     operational complexity for marginal benefit. Confirm or override this with 
     Jacob before implementing.

     Schema sketch (record format) ‚ÄĒ this is the future-proof shape:
     {
       "id": "uuid-v4",
       "schema_version": 1,
       "source": "voice|journal|import|...",
       "source_id": "external identifier if any (mbox msg-id, audio file path,
     etc.)",
       "ingested_at": "ISO-8601 timestamp",
       "content": "the actual text",
       "content_hash": "sha256 of content (for idempotency)",
       "metadata": { "mood": "...", "topics": [...], "speaker": "...", ... },
       "embedding": [0.123, ...]  // optional; lazily computed if missing
     }

     Why each field:
     - schema_version: when we change the shape, old records remain readable;
     migrations are explicit
     - source + source_id: provenance ‚ÄĒ we always know where a record came from
     - ingested_at: temporal grounding for "what was Jacob feeling last Tuesday"
     - content_hash: idempotency key ‚ÄĒ re-ingesting the same content writes
     nothing new
     - embedding separate from content: lets us re-embed with a better model
     without touching the source

     Appendix B ‚ÄĒ Beginner glossary

     - venv ‚ÄĒ A Python "virtual environment." A folder that contains its own
     copy of Python and packages, isolated from the system Python. Lets each
     project have its own dependencies without conflict.
     - conftest.py ‚ÄĒ A pytest config file that runs at test collection time.
     Anything you put in it executes before tests start.
     - fixture ‚ÄĒ In pytest, a reusable setup function. Tests that ask for the
     fixture get whatever it produces.
     - subprocess ‚ÄĒ A separate OS process spawned from the main process. Used in
      tests to isolate code that might hang or crash.
     - idempotent ‚ÄĒ An operation you can run twice and get the same result as
     running it once. Critical for ingestion pipelines.
     - schema ‚ÄĒ The shape of data ‚ÄĒ what fields it has, what types those fields
     are. Code talks to data through the schema.
     - schema_version ‚ÄĒ A number on each record saying "this was written under
     rules version N." Lets you change the rules later without breaking old
     records.
     - embedding ‚ÄĒ A list of numbers that represents a chunk of text in a
     high-dimensional space. Two embeddings close together mean two pieces of
     text are semantically similar.
     - content_hash ‚ÄĒ A fingerprint of the content (sha256). Same content ‚Üí same
      hash. Used as a deduplication key.
     - REPL ‚ÄĒ Read-Eval-Print Loop. An interactive prompt that reads input,
     evaluates, prints, and loops forever. Great for exploration; bad if a test
     calls it accidentally.
     - JSONL ‚ÄĒ JSON Lines. A file where each line is one JSON object. Easy to
     grep, jq, awk, or stream.
     - jq ‚ÄĒ A command-line tool for filtering/transforming JSON. Replaces a lot
     of "ask the LLM to summarize this JSON" calls.
     - bisect ‚ÄĒ Binary search through git history to find which commit
     introduced a bug. Step 1: a commit you know is bad. Step 2: a commit you
     know is good. Git walks you through the midpoints.
     - contract test ‚ÄĒ A test that checks the promise an API makes to its users
     (e.g., "this CLI exits with code 2 if args are missing"), not how it
     happens to be implemented today.
     - defense in depth ‚ÄĒ Multiple layered defenses, each catching a different
     class of failure. The Swiss cheese model: no slice is whole, but the layers
      cover each other's holes.
