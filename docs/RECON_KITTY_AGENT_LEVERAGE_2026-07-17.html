1. Executive recommendation (for Jacob)

Two adoption bets, both small, both reversible:
  1. Plain Pydantic Mission (v1) as the ADR-0017 seam — no new dependency, ~130 lines, verified round-trip and strict-field-reject. Ship as gateway/mission.py.
  2. DSPy for compiled reasoning, starting with one Signature (CompileMission). MIT, LiteLLM-native, GEPA/MIPROv2/SIMBA/Assertions all live in v3.2.1. Isolated to experiments/ until one honest eval on next_step beats hand-tuned within cost budget on the cheap route.
Everything else — LangGraph, MSAF, OpenAI Agents SDK, Letta as runtime, Agent Lightning, TextGrad, PydanticAI as adopted framework — does not survive the evidence. Details in §9.

The one-line verdict on Builder: do not replace it with any workflow framework. LangGraph/MSAF/Temporal solve Kitty's reasoning-graph checkpointing, not Builder's execution checkpointing. Builder already ships execution. See §8.

6. Prototype methodology & results

All prototypes ran in an isolated Python 3.12.13 venv at docs/recon/2026-07-17-agent-leverage/experiments/.venv. Dependency install measured at 255 MB total (litellm 94 M, numpy 34 M, openai 19 M, anthropic 12 M, pydantic-ai 8 M, dspy 3 M, plus transitive). Nothing installed into Kitty's runtime venv.

Prototype 1 — Typed Mission (Pydantic vs PydanticAI)

MEASURED. Two files, both green in this session:

  • proto_1_mission_contract/mission_pydantic.py — 5 assertions pass: JSON round-trip (1134 chars), unknown-field rejection via extra='forbid', bad-SHA rejection via @field_validator, all 10 MissionState enum values, missing-required rejection.
  • proto_1_mission_contract/mission_pydantic_ai.py — Agent(output_type=Mission) constructs, output_validator + ModelRetry register, tool_plain registers; end-to-end structured output via FunctionModel emits a schema-valid Mission and PydanticAI validates it.

Findings:

  • MEASURED. PydanticAI injects the Mission's JSON schema (Mission.model_json_schema()) as prompt content: 5820 chars ≈ 1455 tokens of fixed prompt overhead per structured call. That is a real per-call cost.
  • MEASURED. PydanticAI's default retry budget is 1. First run of Proto 1b threw UnexpectedModelBehavior: Exceeded maximum output retries (1) because TestModel synthesized a mission_id="a" which violated the strict pattern ^[A-Z]+-\d+$. A real LLM would hit the same failure mode occasionally.
  • INFERRED. PydanticAI's incremental value over plain Pydantic (~130 LoC) is: structured-output retry loop + typed tool registration + ModelRetry exception. All of which Kitty can replicate in ~50 lines of custom code. For only 1–2 typed outputs (Mission being the primary), custom wins.

Prototype 4 — LangGraph checkpoint pattern read (no install)

REPO FACT (fetched via gh api). LangGraph checkpoint interface:

  • BaseCheckpointSaver with .put, .get, .list, .put_writes. SQLite/Postgres/in-memory backends. langgraph-checkpoint-sqlite pulls only aiosqlite + sqlite-vec.
  • Concept: thread_id → sequence of checkpoints (superstep snapshots); optional checkpoint_id for mid-thread resume.
  • Pending writes: partial superstep results survive so nodes don't re-run on resume.
  • Security note from the README itself: default JsonPlusSerializer accepts any Python type. Deserialization of untrusted checkpoints could execute code. New apps should set LANGGRAPH_STRICT_MSGPACK=true.

INFERRED. Kitty's context_receipt.py + Mission (with revision/state) + Builder queue already provide equivalent semantics: read-time freshness invariants, mid-flight state, and durable resume — with a hard-fail security model that doesn't include arbitrary deserialization. Nothing to borrow from LangGraph's SQLite schema that improves on Kitty's existing shape.

Prototype 5 — Letta memory-block pattern read (no install)

REPO FACT. Letta memory model (from letta/schemas/memory.py):

  • Memory(blocks: List[Block], file_blocks: List[FileBlock], agent_type, git_enabled).
  • ContextWindowOverview introspects tokens per section: system, core memory, memory filesystem, tool usage rules, directories, summary memory, functions definitions, messages.
  • Runtime shape: SQLAlchemy + alembic + fastapi + uvicorn — server-oriented, not a library.

INFERRED. The Block schema shape (labelled semantic memory sections with per-section token counts) is a useful pattern Kitty's context_assembler.py could formalize. But the runtime (Postgres + web server) is the wrong shape for Kitty's local-first single-user model. Extract the pattern, not the framework. Concrete suggestion: add a small MemoryBlock Pydantic type to memory_graph's output so callers can see "what section is this from and how many tokens." Ships in ~30 LoC.

8. Builder-protection analysis

ADR-0015 §2 and ADR-0017 §"Independence guarantees" define the boundary. Nothing in this recon crosses it. Explicit non-replacements:

  1. LangGraph / MSAF / LlamaIndex Workflows are NOT eligible to replace Builder. They solve reasoning-graph checkpointing (the trace of a single Kitty turn). Builder solves execution checkpointing (attempt state, worktree isolation, lease fencing, SHA-256 verification, publication). Different problems. If a reasoning framework is later adopted, it wraps inside Kitty's turn — never around Builder.
  2. Temporal / DBOS / Hatchet / Prefect stay rejected (July audit R1). Replacing Builder's SQLite queue would kill zero-dependency + local-first + fail-loud. PydanticAI's optional integrations with those (via [temporal]/[dbos]/[prefect] extras) are for its own agents; they do not touch Builder.
  3. OpenAI Agents SDK is out on hard-constraint grounds. Provider lock-in violates model/provider neutrality per Kitty's constraints and North Star §3. Read guardrail source patterns; don't adopt the SDK.

Builder-side capabilities that MUST stay Kitty-owned (from July audit's Strategic Ownership Registry, all reaffirmed):

  • Builder packet lifecycle · initiative model · lease semantics · durable attempts with worktree isolation · evidence ledger (attempt → review → publish SHA-256 chain).
  • Kitty side: memory_graph unified read · D10 privacy boundary · SOUL/voice · life-first ordering · reasoning-policy OODA-VL · resume loop.
