# Kitty — Unified Implementation Plan

*Synthesized from: prior VISION/roadmap drafts (now retired into this doc), superpowers platform notes, free-code feature audit, live `gateway/` tree, PARKED_FEATURES.md, OPEN_LOOPS.md, and specs.*
*Date: 2026-05-13*

---

## North star (condensed)

Kitty is **leverage** for a non-coder who still steers product: a **companion** that can critique, teach in small bites, and eventually run multi-step work without “type continue forever.” Long-form teaching / tone lives in **`docs/COMPANION_VOICE_CHARTER.md`** and **`docs/DECISIONS.md`** (e.g. partner-not-servant framing). This plan is the **execution spine**; sentiment without tasks is not enough.

---

## What Kitty Is (Current Reality)

A FastAPI gateway (`gateway/`) behind Open WebUI + Garage UI:
- LLM routing through LiteLLM with 4-tier fallback (LiteLLM → AgentRouter → OpenRouter → Gemini → NVIDIA)
- Domain-classified prompt selection (soul, repair, health, research, code — 5 templates)
- Memory (Mem0/ChromaDB) + Knowledge ingestion pipeline (Clerk→Librarian→Archivist)
- Journal (ambient + intentional interview + LLM synthesis)
- Morning brief (RSS + TASKS.md + memory → LLM synthesis)
- Weekly mirror (Honcho pattern analysis from traces)
- Self-review drift tracking against SOUL.md
- Parts mode (Skeptic/Champion/Pragmatist/Observer internal council)
- Socratic learning (Knowledge Gates every 5 tool calls)
- Voice pipeline (faster-whisper STT + edge-tts TTS)
- Troubleshooter, vision analysis, inventory photo extraction
- Autonomy state tracking (persistent multi-turn sessions)
- Token usage logging + semantic caching
- Onboarding (8-domain interview, partially implemented)
- Garage UI (Next.js dashboard: Chat/Journal/Evals/Trust views)

**Test count moves with CI — verify with `/opt/homebrew/bin/python3.12 -m pytest tests/ -q` (see root `TASKS.md`). Specialists tree removed from this checkout. External world agents/plugins are roadmap phases, not all live yet.**

## Phase 1: Companion Architecture Foundation

*Weeks 1-2. Ship a reactive but personality-rich Kitty with voice and ambient awareness.*

### 1.1 Collapse Context Layers Into One System

**Problem:** Memory (Mem0), Knowledge (ChromaDB), Journal (JSONL), and Traces (JSONL) live in four separate systems queried separately. `context_builder.py` already fetches memory + knowledge — unify them.

**Changes:**
- `gateway/memory_graph.py` — unified query function that merges results from all four stores
- `gateway/context_builder.py` — swap separate `_fetch_memory` + `_fetch_knowledge` for single `_fetch_context`
- Add journal entries and recent traces to the unified query
- Cross-system query: "What was I working on when I felt frustrated about the Sansui?"

**Files:** `gateway/memory_graph.py` (new), `gateway/context_builder.py` (modify), add tests

### 1.2 Companion Voice — Actually Wire It

**Problem:** The COMPANION_VOICE_CHARTER.md exists but Kitty's personality is purely prompt engineering. The self-review drift tracking exists but doesn't feed back into response generation.

**Changes:**
- `gateway/voice_gate.py` — pre-response filter that checks against SOUL.md violations BEFORE sending
- Wire `self_review.py` drift signals into `context_builder.py` so Kitty knows when she's drifting
- Post-response: if drift detected, append a "correction nudge" to the next system prompt

**Files:** `gateway/voice_gate.py` (new), `gateway/self_review.py` (modify), `gateway/context_builder.py` (modify)

### 1.3 Persistent Voice Channel

**Problem:** Voice is one-shot record→transcribe→text. Not a conversation channel. Jacob works on hardware and needs hands-free interaction.

**Changes:**
- `gateway/voice_session.py` — WebSocket endpoint for long-lived voice sessions
- Push-to-talk or VAD-based (voice activity detection)
- Stateful: maintains conversation context, doesn't restart per utterance
- Garage UI: replace one-shot recording with persistent voice button that stays open

**Files:** `gateway/voice_session.py` (new), `gateway/app.py` (add WebSocket route), `kitty-chat/app/page.tsx` (modify voice toggle)

### 1.4 Buddy/Companion System (port from free-code)

**Problem:** Garage UI's Mascot.tsx is a static SVG component. free-code has a Pokémon-like companion with species/rarity/stats/hats/idle-animations/speech-bubbles.

**Changes:**
- `kitty-chat/app/components/Mascot.tsx` — full rewrite based on free-code's buddy system
- Deterministic species (18 options: cat, duck, goose, dragon, axolotl, capybara, robot, etc.) from user ID hash
- Rarity system (common→legendary with weighted rolls) + stats (DEBUGGING, PATIENCE, CHAOS, WISDOM, SNARK)
- Hats (crown, tophat, propeller, wizard, beanie, tinyduck)
- Idle animation sequence (fidget, blink, rest)
- Speech bubbles with fade-out (free-code's SpeechBubble component)
- Pet interaction (`/buddy pet` → hearts animation)
- Name + personality generated by LLM on first "hatch"
- Mood states driven by `self_review.py` reaction signals (happy when Jacob celebrates, concerned when frustrated)

**Files:** `kitty-chat/app/components/Mascot.tsx` (rewrite), `kitty-chat/app/components/buddy/` (new: companion.ts, sprites.ts, types.ts, SpeechBubble.tsx), `gateway/buddy.py` (new: hatch endpoint, mood sync)

## Phase 2: Agent System & Background Tasks (free-code port)

*Weeks 3-4. Give Kitty the ability to spawn autonomous agents and run background tasks.*

### 2.1 Autonomous Sub-Agent System

**Problem:** Kitty has no way to spawn workers for parallel or background work. Everything is synchronous request-response.

**Changes:**
- `gateway/agent_runner.py` — spawn sub-agents with goals and tool budgets
- Built-in agent presets ported from free-code: `explorer`, `planner`, `coder`, `reviewer`, `researcher`
- AgentTool: model can invoke agents programmatically via `POST /agent/spawn`
- Agent memory snapshots (wire `autonomy_state.py` as the backend)
- Resume/reconnect to running agents (`GET /agent/{id}`, `GET /agent/{id}/output`)
- Agent display in Garage UI: `ActiveNodes.tsx` already tracks nodes — extend to show agents

**Files:** `gateway/agent_runner.py` (new), `gateway/agent_tool.py` (new — model-invokable tool), `gateway/app.py` (add routes), `kitty-chat/app/components/AgentPanel.tsx` (new)

### 2.2 Background Task System

**Problem:** No background task execution. Ingestion queue exists but is just a SQLite list. No general task system.

**Changes:**
- `gateway/task_runner.py` — background task execution system  
- Task types: `research`, `ingest`, `build`, `cleanup`, `dream` (overnight batch)
- `POST /task/create` — create a task with goal + model + tool budget
- Tasks run as subprocess workers with progress reporting via WebSocket
- Task list + output retrieval (`GET /tasks`, `GET /tasks/{id}/output`)
- DreamTask: overnight batch processing (ingestion queue, cleanup, pattern analysis)
- Port free-code's `TaskCreateTool`, `TaskListTool`, `TaskOutputTool`, `TaskStopTool` patterns

**Files:** `gateway/task_runner.py` (new), `gateway/task_models.py` (new — Pydantic task schemas), `gateway/app.py` (add routes), `kitty-chat/app/components/TaskPanel.tsx` (new)

### 2.3 TodoWriteTool — Structured Task Tracking

**Problem:** `gateway/tasks.py` only syncs one line in TASKS.md. No structured task list the model can maintain across turns.

**Changes:**
- `gateway/todo_store.py` — SQLite-backed structured task list
- Tasks with: `id`, `content`, `status` (pending/in_progress/completed/deprioritized), `active_form`
- Model-invokable: `POST /todos` creates/updates list, `GET /todos` retrieves
- Garage UI: `TodoPanel` showing live task list with status indicators
- Port free-code's `TodoWriteTool` pattern (the model writes its own task list)

**Files:** `gateway/todo_store.py` (new), `gateway/app.py` (add routes), `kitty-chat/app/components/TodoPanel.tsx` (new)

### 2.4 Skill System — Runtime-Invokable

**Problem:** Skills exist in `.agents/skills/` and `consolidated-skills/` but can only be invoked via slash commands or agent instructions. No runtime skill discovery or programmatic invocation.

**Changes:**
- `gateway/skill_registry.py` — discover and register skills from disk
- `GET /skills` — returns available skills with descriptions, when-to-use hints, allowed tools, model preference
- `POST /skill/invoke` — invoke a skill programmatically with arguments
- SkillTool pattern from free-code: model can call skills as tools
- Skills can specify context mode (inline vs fork), bundling reference files

**Files:** `gateway/skill_registry.py` (new), `gateway/skill_tool.py` (new), `gateway/app.py` (add routes)

## Phase 3: The Hands — External World Connection

*Weeks 5-6. Give Kitty real-world capabilities. This is Phase 3 from the North star: hands to interact with calendar, files, and messaging.*

### 3.1 Push Notifications (from Phase 7 brief plan)

**Problem:** Kitty can only communicate through the web UI. No phone notifications. Already in the Phase 7 plan.

**Files:** `gateway/notify.py` (new), `gateway/brief.py` (modify), `.env.example` (modify)

### 3.2 Calendar Integration

**Problem:** No calendar awareness — required for the external-world phase above.

**Changes:**
- `gateway/calendar.py` — read/write macOS Calendar via AppleScript (local) or Google Calendar API (cloud)
- `GET /calendar/today` — today's events
- `POST /calendar/event` — create event
- Morning brief includes calendar events (already specced in Phase 7 plan)
- Calendar events feed into context builder: "You have a meeting in 30 minutes"
- Proactive: "You have a 2-hour gap this afternoon — want to work on the Sansui?"

**Files:** `gateway/calendar.py` (new), `gateway/app.py` (add routes), `gateway/brief.py` (modify), `gateway/context_builder.py` (modify)

### 3.3 iMessage Bridge

**Problem:** No messaging integration — part of "hands on the Mac" scope (files, browser, iMessage).

**Changes:**
- `gateway/imessage.py` — send/receive iMessages via AppleScript bridge
- `POST /imessage/send` — send a message
- `GET /imessage/recent` — read recent messages
- iMessage voice corpus ingestion (already specced in VOICE_CORPUS.md and STANDUP.md)
- Proactive: "Maya texted — want me to reply?" (requires polling or notification hook)

**Files:** `gateway/imessage.py` (new), `gateway/app.py` (add routes)

### 3.4 Telegram/Mobile Bridge

**Problem:** Kitty is only reachable on desktop. Platform design spec mentions Telegram as multi-channel access.

**Changes:**
- `gateway/telegram_bot.py` — Telegram bot endpoint (polling or webhook)
- Routes messages through the same Gateway pipeline as web chat
- Voice messages → STT → Gateway → TTS → voice reply
- Commands: `/brief`, `/stuck`, `/journal`, `/tasks`
- Push delivery: agent results, task completions, proactive nudges

**Files:** `gateway/telegram_bot.py` (new), `gateway/app.py` (add webhook route)

### 3.5 Web Monitoring Agent

**Problem:** `researcher.py` does one-shot Tavily search. No persistent monitoring. Jacob needs to track Craigslist/Kijiji listings and forum threads.

**Changes:**
- `gateway/web_monitor.py` — persistent URL polling and change detection
- `POST /monitor/create` — register a URL to watch with a query/filter
- `GET /monitor/status` — list monitored URLs and last check
- Notify via Pushover/Telegram on match
- Built-in site templates: Kijiji, Craigslist, Audiokarma, DIYAudio

**Files:** `gateway/web_monitor.py` (new), `gateway/app.py` (add routes)

## Phase 4: Autonomous Building & Value Generation

*Weeks 7-8. Autonomous execution: multi-step tasks with approval gates, minimal human babysitting (see Phase 1 autonomy + agent work).*

### 4.1 Full Build Pipeline

**Problem:** No spec→scaffold→implement→test→deploy pipeline. `tasks.py` updates one line in TASKS.md. `autonomy_state.py` tracks sessions but doesn't execute builds.

**Changes:**
- `gateway/builder.py` — autonomous build orchestration
- Pipeline stages: spec intake → scaffold → implement → test → self-review → deploy
- Each stage is a sub-agent task via `agent_runner.py`
- Approval gate between stages (user must approve before code changes hit disk)
- Auto-commit with conventional commit messages
- Build history and rollback

**Files:** `gateway/builder.py` (new), `gateway/builder_spec.py` (new — Pydantic spec model), `gateway/app.py` (add routes), `gateway/agent_runner.py` (modify)

### 4.2 Verification Agent (port from free-code)

**Problem:** No automated verification of agent work. Tests exist but aren't wired into the build loop.

**Changes:**
- `gateway/verifier.py` — post-change verification agent
- After any code change: spawn verification sub-agent that runs tests, checks for regressions
- Report: pass/fail, diff summary, performance delta
- Auto-rollback on verification failure
- Port free-code's `VerifyPlanExecutionTool` pattern

**Files:** `gateway/verifier.py` (new), `gateway/builder.py` (modify)

### 4.3 Eval-Gated Development

**Problem:** Evals exist (`tests/test_eval_*.py`, eval dashboard) but aren't tied to the build loop. No before/after comparison.

**Changes:**
- `gateway/eval_runner.py` — run eval suite before and after changes
- Score comparison: did the change improve or degrade quality?
- Auto-block deploy on eval regression
- Eval dashboard already shows results — add trend lines and regression alerts

**Files:** `gateway/eval_runner.py` (new), `gateway/builder.py` (modify), `kitty-chat/app/components/EvalDashboard.tsx` (modify)

## Phase 5: Plugin System & Extensibility (free-code port)

*Weeks 9-10. Make Kitty extensible without touching core gateway code.*

### 5.1 Built-in Plugin Registry

**Problem:** All features are tightly coupled in `gateway/`. No way to add capabilities without editing core files.

**Changes:**
- `gateway/plugin_registry.py` — port free-code's `plugins/builtinPlugins.ts` pattern
- Register plugins with: name, description, version, skills, hooks, MCP servers
- User-toggleable enable/disable persisted to settings
- Plugin marketplace concept (local-only initially, JSON file listing available plugins)
- `/plugin` management commands in chat and API

**Files:** `gateway/plugin_registry.py` (new), `gateway/plugin_loader.py` (new), `gateway/app.py` (add routes)

### 5.2 MCP Tool Integration (port from free-code)

**Problem:** MCP servers are configured in `.mcp.json` but not invokable at runtime by Kitty.

**Changes:**
- `gateway/mcp_tool_bridge.py` — runtime MCP tool discovery and invocation
- `MCPTool` — model-invokable: list MCP servers, discover tools, invoke with arguments
- Port free-code's `MCPTool`, `ListMcpResourcesTool`, `ReadMcpResourceTool`
- This unlocks: file system MCP, memory MCP, and future external service MCPs
- Safety: tool allowlisting per plugin/skill

**Files:** `gateway/mcp_tool_bridge.py` (new), `gateway/mcp_tool.py` (new), `gateway/app.py` (add routes)

### 5.3 Cron/Trigger System (port from free-code)

**Problem:** Scheduled tasks use launchd plists (morning brief, doctor). No runtime scheduling. No user-configurable triggers.

**Changes:**
- `gateway/cron.py` — runtime cron scheduler
- `ScheduleCronTool` pattern: model or user can schedule recurring tasks
- Triggers: time-based (daily at 7am), event-based (on session end), condition-based (when memory size > X)
- Port free-code's `AGENT_TRIGGERS` with `ScheduleCronTool`
- Jitter to avoid thundering herds
- Replace launchd plists with runtime-managed schedules

**Files:** `gateway/cron.py` (new), `gateway/cron_tool.py` (new), `gateway/app.py` (modify startup)

## Phase 6: Intelligence Depth

*Weeks 11-12. Make Kitty genuinely smart about Jacob's life patterns.*

### 6.1 Apple Health Ingestion (from platform design spec)

**Problem:** No health data integration. Platform design Phase 6 explicitly calls for Apple Health export parsing.

**Changes:**
- `gateway/health_parser.py` — dedicated Apple Health XML parser
- Extracts: sleep, resting heart rate, weight, steps, HRV, workouts
- Aggregates into weekly JSON summaries (NOT raw XML ingestion — that's 500MB+)
- Ingested into ChromaDB with `sensitivity: medical`, `allowed_models: [local_only]`
- Feeds morning brief: "Your sleep's been trending down this week"
- Feeds fitness domain: "You've hit the gym 3x this week — that's your best streak in a month"

**Files:** `gateway/health_parser.py` (new), `gateway/ingestion_queue.py` (modify — add health source type)

### 6.2 OBD/Vehicle Data Integration (from platform design spec)

**Problem:** OBD parser referenced in platform design as existing (`src/tools/obd_parser.py`) but not wired into the active gateway.

**Changes:**
- `gateway/obd_reader.py` — OBD-II data ingestion from log files
- Structured vehicle health data: DTCs, sensor readings, fuel trim
- Feeds automotive domain: "Your Ridgeline's fuel trim is trending lean — might be a vacuum leak"
- Proactive maintenance reminders based on mileage patterns

**Files:** `gateway/obd_reader.py` (new or port from archive)

### 6.3 Longitudinal Pattern Tracking

**Problem:** Weekly mirror (`honcho.py`) looks at 7 days. No year-over-year perspective. No life timeline.

**Changes:**
- `gateway/patterns.py` — multi-timeframe pattern detection
- Weekly, monthly, quarterly, annual summaries
- Detect: productivity cycles, mood seasons, recurring frustrations, growth arcs
- Annual review generation: "Here's what changed in 2026"
- Life timeline visualization data (for Garage UI timeline component)

**Files:** `gateway/patterns.py` (new), `gateway/honcho.py` (modify)

### 6.4 Full File Ingestion Pipeline (from platform design Phase 6)

**Problem:** Platform design Phase 6 calls for full ingestion: session logs, chat exports (Claude.ai, ChatGPT), calendar, OBD, Apple Health. Most of this isn't wired.

**Changes:**
- Claude.ai export parser (JSON → structured memory)
- ChatGPT export parser (JSON → structured memory)
- Calendar (.ics) parser
- All with privacy labels and sensitivity routing
- Idempotent: hash-check before re-ingesting
- `gateway/ingestion_queue.py` extended with source-specific parsers

**Files:** `gateway/chat_import.py` (new — Claude/ChatGPT export parsers), `gateway/ingestion_queue.py` (modify)

## Phase 7: Ambient Intelligence

*Weeks 13-14. Kitty notices things without being asked.*

### 7.1 Proactive Nudge Engine

**Problem:** Kitty is 100% reactive. Never initiates. The Companion Voice Charter says "notice it, don't shame it" — but she can't notice if she never initiates.

**Changes:**
- `gateway/nudge.py` — proactive pattern detection and notification
- Triggers:
  - Repeated research without action (3x same topic → "Want me to just find the answer?")
  - Dropped threads (topic mentioned, then silence for days → "Still thinking about that bias circuit?")
  - Mood shifts (journal entries trending negative → gentle check-in)
  - Milestone moments (first completed build, health streak → celebration nudge)
  - Time-based (calendar gap → "Free afternoon — Sansui time?")
- Delivery: Garage UI banner, Pushover notification, Telegram message
- Configurable throttle per domain and time-of-day

**Files:** `gateway/nudge.py` (new), `gateway/nudge_rules.py` (new — configurable rule engine)

### 7.2 Ambient Context Awareness

**Problem:** No awareness of what Jacob is doing. Can't preload context or auto-switch modes.

**Changes:**
- `gateway/ambient.py` — opt-in local context detection
- Active application detection (macOS AppleScript)
- File system watch: what's Jacob editing?
- Auto-mode switching: working on PCB photo → auto-switch to repair domain
- Predictive context preloading: about to work on Sansui → preload service manual chunks
- All local-only, no cloud, fully opt-in with kill switch

**Files:** `gateway/ambient.py` (new)

### 7.3 Context Compaction (port from free-code)

**Problem:** Long conversations fill the context window. No smart summarization before overflow.

**Changes:**
- `gateway/context_compactor.py` — port free-code's `COMPACTION_REMINDERS`
- Auto-detect context window pressure
- Smart summarization before overflow: compress history, keep key facts
- User notification: "Context getting full — compacted the last 20 messages, kept the important bits"
- Cached compaction state for fast resume

**Files:** `gateway/context_compactor.py` (new), `gateway/context_builder.py` (modify)
