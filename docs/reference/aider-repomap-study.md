# Aider Repomap Study

**Date:** 2026-07-14
**Status:** Reference — study only
**Source:** [Aider-AI/aider](https://github.com/Aider-AI/aider) (47.4k stars, Apache 2.0, Python)

## Purpose

Aider generates a "repository map" (repomap) — an LLM-optimized summary of the
codebase structure — to give coding agents context about the project before they
edit files. This document compares Aider's approach with Kitty's existing code
knowledge model and identifies improvements for Kitty's agent context assembly.

## How Aider's Repomap Works

Aider's repomap (`aider/repomap.py`) generates a concise structural overview:

1. **Tree-sitter parsing**: Parses every file in the repo using tree-sitter to
   extract function signatures, class definitions, type signatures, and import
   relationships.

2. **Token budget**: Targets a configurable token budget (default ~1000 tokens)
   for the entire map, making it suitable for inclusion in LLM context windows.

3. **Relevance ranking**: Files are ranked by relevance to the current task
   using git history (recently modified files rank higher) and structural
   importance (files with many dependents rank higher).

4. **Symbol-level detail**: Only top-level symbols are included. Implementation
   details are omitted. The output looks like:

   ```
   gateway/builder_queue.py:
     QUEUED = "queued"
     CLAIMED = "claimed"
     ...
     def create_task(title, ...) -> dict
     def claim_task(task_id, worker_id, ...) -> dict
     class LeaseConflictError(ValueError): ...

   gateway/memory_graph.py:
     class Item: ...
     class MemoryGraph:
       def search_all(message) -> GraphResult
   ```

5. **Idempotent**: The repomap is regenerated on every request. No caching
   (tree-sitter parsing is fast enough for most repos).

## Kitty's Current Approach

Kitty has three layers of code knowledge:

| Layer | Tool | What It Provides |
|---|---|---|
| **Structural** | `.codegraph/` | SQLite knowledge graph of symbols, edges, files. Queries return symbol source + call paths. |
| **Conceptual** | `docs/codemap/` | Handwritten documents explaining the codebase at the architectural level. Five lenses: overview, capabilities, dataflow, codemap, domain. |
| **LLM guidance** | `CLAUDE.md`, `AGENTS.md`, `docs/ARCHITECTURE.md` | Prose documents telling agents how to navigate the repo. |

## Comparison

| Dimension | Aider Repomap | Kitty (codegraph + codemap) |
|---|---|---|
| **Granularity** | Symbol-level (functions, classes, imports) | Symbol-level (codegraph) + concept-level (codemap) |
| **Token efficiency** | Configurable budget, ~1000 tokens | No budget — codegraph returns full source of matched symbols |
| **Freshness** | Generated on every request from live AST | codegraph index updated by daemon; codemap may drift |
| **Relevance** | Ranks by git recency + dependency count | codegraph ranks by query relevance; codemap is static |
| **LLM integration** | Designed to be prepended to system prompt | codegraph used for exploration queries; codemap for human reading |
| **Scope** | Full repo structural overview | Targeted symbol/flow queries |

## What Kitty Should Adopt

### 1. Token-budgeted structural summary for agent context

**Problem:** Kitty's Builder packets include context bundles (`builder_attempt.py`
context manifest) but these don't include a structural summary of the repo.
Agents working in isolated worktrees see the full file tree but lack a
compressed overview.

**Recommendation:** Add a `repo_map` field to the context bundle generated by
`builder_context.py`. The map should be a ~500-token tree-sitter summary of
the files the packet is allowed to touch (from `allowed_paths`). This gives
the worker agent a structural overview before it starts editing.

### 2. Git-recency-based file ranking

**Problem:** Kitty's codegraph returns all matching symbols regardless of when
they were last changed. For a refactoring task, the agent should see recently
modified files first.

**Recommendation:** When building the context map for a packet, sort files by
`git log --oneline -- <path>` recency. Recently touched files are more likely
to be relevant to the current work.

### 3. Tree-sitter as a lightweight alternative to codegraph for simple queries

**Problem:** codegraph requires a running daemon and an indexed database. For
quick "what symbols are in this file" queries, tree-sitter is faster and has
no setup cost.

**Recommendation:** Use tree-sitter (already available via ast-grep in Kitty's
toolchain) for simple symbol extraction in context bundles. Reserve codegraph
for cross-file call path queries.

## What Kitty Should Keep

### 1. Multi-layer knowledge model

Aider's repomap is purely structural. Kitty's three-layer model (structural
via codegraph, conceptual via codemap, guidance via CLAUDE.md) gives agents
a richer understanding. **Keep all three layers** — they serve different
purposes.

### 2. ADR integration

Kitty's ADRs are part of the agent context. Aider has no equivalent. The
ability to see *why* a module was designed a certain way (via the ADR) is
a superpower that no structural tool provides.

### 3. Packet context model

Kitty's packet-based delegation (objective, allowed paths, acceptance criteria,
prior attempts) gives agents a bounded task scope. Aider works on the full
repo. **Keep the packet model** — it prevents scope creep.

## Concrete Improvements

1. **Add `repo_map` to context bundles** in `builder_context.py`:
   A 500-token structural summary of the packet's `allowed_paths`, generated
   via tree-sitter symbol extraction.

2. **Sort context files by git recency** in `builder_context.py`:
   Use `git log --diff-filter=M --oneline` to rank files.

3. **Document the knowledge model** in `docs/reference/repo-knowledge-model.md`:
   Explain the three-layer model (structural, conceptual, guidance) so future
   contributors understand why it exists and how to maintain it.

## Verdict

Kitty's three-layer knowledge model is stronger than Aider's single-layer
repomap for agent context. The main gap is that Kitty doesn't include a
compressed structural summary in worker context bundles — adding one would
bring the best of both approaches together.

Do not replace codegraph or codemap with Aider's repomap. Instead, add a
lightweight repomap-like summary to the existing context bundle pipeline.
